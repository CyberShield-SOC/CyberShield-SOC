# Frontend architecture

## Design goals

The application separates screen composition from data access so backend integration and future production iterations do not require a UI rewrite. Route pages receive normalized domain objects from a repository contract. Shared shell, panel, badge, chart, table, loading, and error primitives keep visual and accessibility behavior consistent.

## Layers

1. `frontend/src/main.jsx` installs the application-level render boundary, and `frontend/src/App.jsx` selects the public authentication experience or lazy-loaded SOC application.
2. `frontend/src/hooks/useAuthRoute.js` owns allow-listed hash routing. Unknown routes safely return to login.
3. `frontend/src/soc/SocApp.jsx` maps allowed SOC routes to lazy-loaded feature pages.
4. `frontend/src/soc/context/SocWorkspaceContext.jsx` owns synchronized alerts, incidents, events, notes, notification preferences, workspace settings, counts, refresh state, and optimistic mutations.
5. `frontend/src/soc/components/` provides the application shell and reusable visual primitives.
6. `frontend/src/soc/pages/` owns route-local filters, drafts, and presentation. Cross-route alert, incident, and event selection lives in the workspace provider.
7. `frontend/src/soc/services/socRepository.js` exposes the data contract and selects a mock or HTTP adapter.
8. `frontend/src/soc/data/` contains development fixtures only.

`frontend/src/soc/utils/incidentWorkflow.js` is the single source of truth for the supported incident statuses, terminal-state detection, display labels, and next workflow action. Pages and repository adapters must use these helpers instead of recreating status arrays or comparing raw strings.

## Adding a feature page

1. Add the allow-listed route to `SOC_ROUTES`.
2. Add the navigation entry and icon to `NAV_GROUPS` in `SocShell.jsx`.
3. Add a lazy route component in `SocApp.jsx`.
4. Compose the page from `PageHeader`, `StatCard`, `Panel`, badges, charts, and tables.
5. Add repository methods and backend-contract documentation before calling a new endpoint.
6. Cover pure filtering, mapping, validation, or export logic with a Node test.

## Data and state

`SocWorkspaceContext` loads each server resource through the repository contract and exposes consistent loading, error, refresh, and mutation state to route pages. A per-resource request version prevents a slower, older refresh from replacing newer data. Local page state is reserved for transient filters, drafts, and mock-mode demonstrations. The analyst's explicit alert, incident, and event selections are shared between Alerts, Dashboard, Quick Resolve, Event Logs, Notes, and Incidents so route changes and table pagination do not discard the active investigation. Notifications carry an allow-listed record type plus record identifier and set the matching shared selection before navigating. The connected user's display name is exposed as `currentActor` and is used for optimistic attribution instead of a hard-coded analyst name. These values remain memory-only and reset on sign-out. If the backend later justifies a query-cache library, it can replace the provider's loading layer without changing page markup because access remains behind the repository.

Concurrent dashboard, event, alert, and incident refreshes share identical in-flight API reads. The deduper never caches a completed response, so a later manual refresh still reaches the server while initial workspace loading avoids downloading and normalizing the same payload multiple times.

The shell-owned global time range is normalized in the workspace provider and exposes filtered event, alert, and incident collections. Queue pages use the event observation timestamp, while ingestion cards/charts use the persisted `ingested_at` timestamp; this keeps an old event uploaded today from being presented as newly observed. The All available range derives stable buckets from the available timestamps, and a successful upload selects it so historical evidence is not hidden by a recent-time filter. Navigation counts, dashboard cards, and charts consume the appropriate derived collections. `utils/timeRange.js` contains deterministic bucket, severity, grade, and source-IP aggregation functions. `utils/ipLocation.js` classifies private and documentation ranges, consumes country metadata only when telemetry supplies it, and retains unknown addresses in Top Sources instead of silently dropping them. Invalid legacy display-only timestamps stay visible in queues but are excluded from interval buckets.

API health is a separate polled resource with a 60-second interval and an independent manual refresh. Polling pauses while the document is hidden and refreshes when the tab becomes visible, reducing background traffic without leaving a returning analyst with stale status. The sidebar exposes the same state through color, text, and an accessible live status: green/Operational, yellow/Refreshing, or red/Offline. Its status does not replace route error handling or backend monitoring.

Mock-mode incident identifiers come from `utils/recordIds.js`, which advances from the highest known identifier instead of relying on array length or timestamps. The connected API is authoritative for incident identifiers. Mock-mode notes and settings are shape-checked before restoration; connected notes are incident-scoped PostgreSQL records loaded through one bounded `GET /notes` request, while settings remain tab-scoped until the backend adds a settings resource.

Do not place session tokens or sensitive evidence in local storage. The visual theme is the only local-storage value. The demo session, mock analyst notes, notification preferences, and settings drafts use tab-scoped session storage and are not a production persistence mechanism.

## Route and session boundary

`useSession` selects the 30-minute tab-scoped demonstration guard or the connected FastAPI session. Connected mode checks `/auth/me` before allowing a protected route, returns rejected sessions to login, shows one concise expiration notice after a previously authenticated protected request returns `401`, and revokes the server session on logout. FastAPI continues to authorize every request independently of the interface guard.

`AppErrorBoundary` is the final render and lazy-load fallback. It presents a neutral reload action without exposing component stacks or evidence. Repository errors remain route-local so one failed resource does not replace the entire application.

## Charts

Charts are lightweight SVG/CSS React components with accessible labels. Connected mode derives the compact KPI strip, event timeline, severity distribution, top-source ranking, evidence-backed Threat analysis, and operational-readiness cards from persisted alerts, incidents, and the latest upload because the Sprint 1 backend has no dashboard endpoint. `utils/dashboardMetrics.js` owns the deterministic threat-priority and readiness logic so presentation markup does not invent findings. The event timeline uses a truthful zero baseline, rounded count scale, labeled value axis, synchronized range selector, date-aware X-axis intervals, hover or touch inspection, and Left/Right/Home/End keyboard navigation with live screen-reader announcements. `utils/chartUtils.js` isolates scale and geometry normalization so empty, zero-only, malformed, single-point, and large-count inputs can be regression tested without a browser. Chart components sanitize non-finite values and render an explicit empty state when there are no points. If a future aggregate endpoint or charting library is added, preserve these component APIs so pages remain stable.

Timestamp extrema use iteration instead of spreading telemetry arrays into `Math.min` or `Math.max`; this avoids engine argument-limit failures on production-sized log imports.

## Detection rule pack and evidence identifiers

`src/soc/data/detectionRulePack.js` mirrors the six rules currently registered by the backend `DetectionEngine`: `R-101` / `brute_force_login` (five failures in 60 seconds), `R-102` / `invalid_user_enumeration` (three distinct usernames in 600 seconds), `R-103` / `sudo_failure` (three failures in 300 seconds), `R-104` / `password_spraying` (five distinct source IPs against one username in 600 seconds), `R-105` / `credential_stuffing_success` (a failure burst followed by a success from the same IP within 120 seconds), and `R-106` / `port_scan` (ten scan events from one source IP in 60 seconds). The Threat Detection page shows only these executable rules, derives match counts from persisted alerts, and remains read-only until the backend exposes an authorized rule-management endpoint. When backend rule registration or defaults change, update this catalog and its focused contract test in the same change.

Persisted logs retain their numeric database identifier as `backendId`. The interface presents that record as `EVT-<id>` and uses the same display identifier for selections, notes, incident evidence, exports, and cross-route navigation. During upload, alerts persist the matched source line numbers; the repository combines upload ID plus those line numbers to resolve the exact stored events. A fallback containing the upload prefix and line number is used only when an API response has not supplied a database ID.

## Styling

`frontend/src/soc/soc.css` translates the Figma design-system tokens into semantic custom properties. Light and dark themes change token values rather than duplicating component rules. The dashboard uses six compact, navigable KPI cards followed by a three-panel analytics row; semantic icon colors and low-opacity under-card highlights communicate state without decorative top strips. Readiness cards use a restrained state-colored border glow, with a stronger treatment reserved for true critical states. The event timeline places dates and times directly on the readable X-axis instead of repeating them in a separate range rail. The evidence-backed threat card uses the selected finding's severity token, while shared operational cards retain one-pixel hover elevation and restrained panel-header washes. Breakpoints collapse the KPI strip, analytics, threat metadata, and action rows for tablet and narrow mobile layouts, and reduced-motion preferences are respected.

Data-heavy tables use bounded two-axis scroll regions instead of widening the document. Alerts, incidents, and event logs share deterministic ten-record pagination with previous/next controls, debounced direct page entry, and automatic reset when filters change. The text-based numeric page field avoids browser-specific spinner controls and commits after a short pause, on blur, or with Enter. Primary and secondary cell content may occupy two lines before truncation, while identifiers, statuses, and timestamps remain stable single-line anchors. Sidebar alert and incident badges display up to `99`, then use `99+` without losing the exact count in the accessible navigation label. Tall desktop detail columns use viewport-aware, keyboard-focusable vertical scroll regions with visible themed scrollbars; below the workspace breakpoint they return to normal document flow so there is only one page scroll. Panel headers, action groups, notices, status controls, modal actions, and long backend-provided values wrap instead of clipping. Modals and notification popovers are capped with dynamic viewport units and safe-area padding, keeping their final controls reachable in short windows, mobile browser chrome, and browser zoom.

Status badges use semantic workflow tokens rather than one generic state color: Open is blue, Investigating is sky blue, Resolved is green, False Positive is neutral, review and approval states are amber, and failure or escalation states are red. Medium severity uses a restrained mustard-yellow token with theme-specific contrast instead of the brighter warning yellow. The base badge rule supplies a neutral fallback so a new backend status remains legible until a deliberate semantic mapping is added. Each custom status token has a light and dark value.

The shared `BrandLogo` component owns the frameless shield fit and size variants so authentication and application-shell branding cannot drift. A shield-shaped CSS clip removes the supplied JPG's rectangular edge without altering the source asset. The sidebar profile trigger exposes its menu state with `aria-expanded`; the same state rotates the chevron upward while the menu enters from its bottom anchor. The `--soc-focus-ring` token gives the composite global-search container one restrained focus treatment in both themes; its nested input and submit icon intentionally do not draw competing outlines. Other form controls use a simple border-color change, and composite filter or assistant fields avoid a second inner focus rectangle.

Operational feedback uses the shared `InlineNotice`, `ValidationMessage`, and `ErrorState` primitives. Complex SOC forms opt out of browser-native validation bubbles and use tested React validation with `aria-invalid`, described error text, and first-invalid-field focus. Shared authentication and SOC invalid-state selectors guarantee a single error border without a competing focus outline or shadow. The backend remains authoritative for every submitted value.

Queue/detail screens resolve only an explicitly selected identifier against the authoritative loaded collection. Filters change the table without silently replacing or discarding the active investigation; missing and stale note links render an unavailable state instead of a blank detail. Search values are bounded, empty collections have explicit states, downloads defer object-URL cleanup until the browser starts the transfer, and upload files are rejected before transmission when their extension, size, or content is invalid. The local and connected inspectors share a 10 MB allow-list for `.log`, `.csv`, `.json`, and `.jsonl`. Backend parsing supports structured CSV, JSON/JSONL, and detected text formats; zero-record parses are rejected without changing the latest dataset. A single workspace mutation lock prevents duplicate or conflicting writes while optimistic failures refresh authoritative server state.

Settings are split by enforcement boundary. Workspace name, default time range, notification-center routing, and AI availability are browser-session preferences with visible effects. The Appearance section adds theme, compact/comfortable/spacious density, small/standard/large workspace-wide text sizing, increased contrast, and reduced-motion controls. Appearance values are allow-listed during restoration so stale or edited storage cannot create unsupported layout states. Theme changes persist through the existing theme hook; other interface preferences use the settings adapter. The light palette uses a bright neutral canvas, white elevated surfaces, a light navigation rail, and visible light scroll tracks; the login artwork receives a brighter aqua treatment while retaining contrast for its white security copy. Its live-event console uses a white frosted surface with stronger framing and larger event type, feature-icon tiles use a darker inset surface for glyph contrast, and light-navigation hover states carry a stronger teal fill. Compact KPI cards use restrained accent borders and a small severity-aware bottom glow, while Threat Analysis, Telemetry Readiness, and Analyst Workload use stronger accent-aware depth to preserve their operational priority. Login composition sizing also steps up on desktop viewports at least 1800px wide and 900px tall so the split layout remains balanced on 1440p-class displays. Session/MFA policy, retention, and data masking remain read-only capability states until backend contracts can enforce them across every route and export.

The authenticated `help` route is lazy-loaded and available to every application role. The top-bar question-mark control opens this Help Center rather than Settings. Help topics link to real application routes, filter locally without transmitting the search term, describe role boundaries, and report whether the workspace is connected to the API or running sample data. Settings also exposes a Documentation section that links into the Help Center while keeping repository deployment and backend-contract documents separate.

Connected workflow controls expose only the four statuses supported by the FastAPI enum: Open, Investigating, Resolved, and False Positive. Incident Tracking summarizes those values as three evenly distributed milestones—Open, Investigating, and Outcome recorded—while the status control preserves the exact terminal outcome. Incident selection is shared between Alerts, Event Logs, the incident queue, Quick Resolve, and tracking so linked-record navigation opens the exact investigation. A linked terminal incident opens its history detail as a one-shot navigation result; dismissing that detail clears the selection, so returning to Incidents cannot resurrect a stale modal. Resolved and False Positive outcomes record the backend's authoritative completion time plus `updated_by_user_id`; the normalized frontend model exposes these as `completedAt`, `completedByUserId`, and a human-readable `completedBy` label in history and workflow views. Playbook checkboxes remain clearly labeled session-only until a backend task contract exists. Analyst notes can be deleted after confirmation, and incident note creation is capped at five in both the UI and API. Viewer accounts receive read-only controls, Analyst accounts can operate alerts/incidents/notes/uploads, and Admin-only routes/settings are guarded in the UI. These controls are usability safeguards; FastAPI role dependencies remain the security boundary.

Manage uses live workspace resources for operational cards that open the supported ingestion, incident, access, and settings workflows. Integrations groups persisted events by telemetry source, supports source filtering and selection, and refreshes authoritative data; provisioning remains explicitly read-only until an authorized integration endpoint exists. Users is a protected Admin-only route backed by list/create endpoints plus atomic profile/access updates, password reset, session revocation, and the compatibility role/enabled-state routes. The Create and Edit forms validate identity and role locally, while password fields are isolated in write-only forms and are never retained by the list or mock adapters. Password reset and account disablement revoke active sessions, and the backend prevents removal of the final active administrator. Other secondary configuration pages report current capability state rather than offering fake switches. A configuration becomes editable only after a repository method and an authorized persistence endpoint exist. See `WORKFLOW_VALIDATION.md` for the interaction and production-readiness matrix.

Alert and incident tables do not infer a row selection when filters or pagination change. Alert details remain pinned to the last explicitly chosen record across pagination and route changes, while a new selection still requires a row or searchable Quick Resolve picker interaction. Alert details explicitly present title, severity, source IP, affected user, rule, reason, observation range, evidence, and type-specific recommended actions. Recommendations are allow-listed by known rule family; unclassified alerts display an analyst-referral state instead of invented guidance. Resolved and False Positive incidents are removed from the active queue and exposed through a searchable, bounded history modal. History export includes completion attribution and uses the same formula-safe CSV escaping as event exports. The `quick-resolve` route composes existing alert status, incident creation/status, incident-note, evidence, rule, and AI-analysis adapters into one analyst workspace; its searchable alert picker matches IDs, titles, IPs, users, rules, severities, and statuses without introducing a parallel data store or bypassing permission checks.

Viewport-bound detail columns and other page-level scroll regions use native vertical scroll chaining (`overscroll-behavior-y: auto`). When an inner region reaches its top or bottom, continued wheel or trackpad input moves the document, matching the common browser behavior used by data-heavy web applications. Horizontal table gestures remain contained so wide records do not accidentally navigate or shift the page. Modal and notification scrolling remains contained because those overlays must not move the obscured page underneath.

Feedback, modal entrances, and incident lifecycle progression use short opacity/position/scale transitions only. The existing `prefers-reduced-motion` rule reduces these animations with the rest of the workspace motion system.

When adding visual states, prefer a semantic custom property over a component-specific hard-coded color. Define the light and dark values together, then consume the token in the shared component rule so future theme changes remain localized.
